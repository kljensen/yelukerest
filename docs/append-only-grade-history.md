# Append-only Grade History Design

## Why

`data.assignment_grade`, `data.quiz_grade`, and `data.grade` currently store one
mutable row per grade. Faculty writes can update or delete those rows in place,
and the Python quiz grader uses `ON CONFLICT DO UPDATE` against
`data.quiz_grade`. That keeps current reads simple, but it means regrades,
corrections, import mistakes, and grade disputes cannot be reconstructed from
database history.

The goal is to make grade changes append-only while preserving the current
student and admin read surface: Elm and PostgREST clients should still read one
current row per assignment submission, quiz/user, or grade snapshot/user.

## Research Notes

- PostgreSQL trigger functions expose `NEW` and `OLD`, and the documentation
  includes audit examples that insert old/new row data into separate audit
  tables on `INSERT`, `UPDATE`, and `DELETE`.
- PostgreSQL views can stay simple for reads. Automatically updatable simple
  views are limited; current-grade views over event history will need
  `INSTEAD OF` triggers for compatible writes.
- PostgreSQL row-level security is evaluated before normal user conditions,
  defaults to denying access when enabled without policies, and is bypassed for
  some referential-integrity checks. We should keep current-row policies simple
  and explicit.
- PostgREST treats the database schema, grants, views, triggers, and RLS
  policies as the API contract, so compatibility is mostly a database design
  problem rather than a frontend problem.

Sources:

- https://www.postgresql.org/docs/current/plpgsql-trigger.html
- https://www.postgresql.org/docs/current/sql-createview.html
- https://www.postgresql.org/docs/current/ddl-rowsecurity.html
- https://docs.postgrest.org/en/stable/references/api/tables_views.html#views

## Current Contract

Current grade storage:

- `data.assignment_grade`: one mutable row per `assignment_submission_id`.
  Defaults fill `assignment_slug`, `points_possible`, and `updated_at`.
- `data.quiz_grade`: one mutable row per `(quiz_id, user_id)`. Defaults fill
  `points_possible`, default `user_id` from `request.user_id()`, and
  `updated_at`.
- `data.grade`: one mutable row per `(snapshot_slug, user_id)` for published
  grade snapshots.

Current API surface:

- `api.assignment_grades`, `api.quiz_grades`, and `api.grades` are `select *`
  views over the data tables and are owned by `api` so table RLS applies.
- Students and TAs can select visible rows. Faculty can select, insert, update,
  and delete through the API views.
- `api.assignment_grade_distributions` and `api.quiz_grade_distributions`
  aggregate current grade rows and intentionally are not owned by `api`, so
  students can see full class distributions rather than only rows visible
  through grade RLS.
- Elm reads `/rest/assignment_grades`, `/rest/quiz_grades`,
  `/rest/assignment_grade_distributions`, and
  `/rest/quiz_grade_distributions`. It does not write grades.
- The in-repo Python quiz grader writes directly to `data.quiz_grade` and
  overwrites grades with `ON CONFLICT DO UPDATE`.

Compatibility constraints:

- Preserve REST resource names and JSON fields for current grades.
- Preserve one-current-row semantics:
  - assignment current row keyed by `assignment_submission_id`
  - quiz current row keyed by `(quiz_id, user_id)`
  - snapshot current row keyed by `(snapshot_slug, user_id)`
- Preserve visible fields and types used by Elm and admin clients, including
  `points`, `points_possible`, `description`, `created_at`, and `updated_at`.
- Preserve distribution columns and compute them from current grades only.
- Preserve existing permission shape unless an implementation issue explicitly
  changes it.

## Proposed Data Model

Add event tables as the source of truth:

- `data.assignment_grade_event`
- `data.quiz_grade_event`
- `data.grade_event`

Each table should have:

- `id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`
- the natural key for the grade:
  - `assignment_submission_id` for assignment grades
  - `(quiz_id, user_id)` for quiz grades
  - `(snapshot_slug, user_id)` for grade snapshots
- denormalized compatibility fields needed for constraints and reads:
  - `assignment_slug` and `points_possible` for assignment grades
  - `points_possible` for quiz grades
- `points REAL NOT NULL`
- `description TEXT`
- `event_type TEXT NOT NULL`, constrained to a small set such as
  `recorded`, `corrected`, and `voided`
- `created_at TIMESTAMPTZ NOT NULL DEFAULT current_timestamp`
- `created_by_user_id INT NULL REFERENCES data.user(id)`
- `source TEXT NOT NULL DEFAULT 'api'`
- `reason TEXT`
- `artifact_uri TEXT`
- `artifact_sha256 TEXT`
- `import_id TEXT`
- `supersedes_event_id BIGINT NULL REFERENCES same_event_table(id)`

Keep the same point-range and foreign-key invariants that current tables have.
The event row should be self-contained enough to explain the grade at the time
it was recorded. Do not depend on mutable current tables to recover basic grade
facts later.

Event tables should be append-only at the privilege boundary:

- Application roles do not receive `UPDATE` or `DELETE` on event tables.
- Faculty/admin changes append a new event.
- A correction appends `event_type = 'corrected'`.
- A removal appends `event_type = 'voided'`; it does not delete earlier events.

## Current-grade Views

Create current views over the event tables:

- `data.assignment_grade_current`
- `data.quiz_grade_current`
- `data.grade_current`

Each current view should return exactly the columns exposed today by the
corresponding current table. Use the newest event per natural key, then hide the
row when that newest event has `event_type = 'voided'`. Do not select the
newest non-voided event, because deleting a grade must not resurrect an older
grade as current. `DISTINCT ON (...) ORDER BY ..., created_at DESC, id DESC` is
a good fit because the natural keys are simple and deterministic.

Timestamp compatibility:

- `created_at` should represent the first non-voided event time for the natural
  key, so existing clients can still treat it as grade creation time.
- `updated_at` should represent the selected current event time, so regrades are
  visible without exposing history.

Then repoint the existing API views:

- `api.assignment_grades` selects from `data.assignment_grade_current`
- `api.quiz_grades` selects from `data.quiz_grade_current`
- `api.grades` selects from `data.grade_current`

Distributions should also read from the current views, not directly from event
tables. They should not count voided or superseded events.

## Compatible Writes

Because the current views will be complex views, preserve the write contract
with `INSTEAD OF` triggers on the API views:

- `INSERT` into `api.assignment_grades`, `api.quiz_grades`, or `api.grades`
  appends a `recorded` event.
- `UPDATE` appends a `corrected` event for the same natural key.
- `DELETE` appends a `voided` event for the same natural key.

Default-filling currently done by `fill_assignment_grade_defaults()` and
`fill_quiz_grade_defaults()` should move into event insertion functions/triggers:

- assignment events fill `assignment_slug` from the submission and
  `points_possible` from the assignment.
- quiz events fill `points_possible` from the quiz and may default `user_id`
  from `request.user_id()` only where the old API allowed that.
- all event insertions validate point ranges and required submissions.

The in-repo Python quiz grader should stop writing `data.quiz_grade` directly.
After the event model exists, it should write through `api.quiz_grades` or a
purpose-built `api.record_quiz_grade(...)` RPC that appends events. Direct
`data.*` writes from the external course admin repo should follow the same
pattern when that work moves into Yelukerest.

## History API

Expose history through separate API views rather than expanding current-grade
responses:

- `api.assignment_grade_events`
- `api.quiz_grade_events`
- `api.grade_events`

Initial permissions should be conservative:

- Faculty can select all event history.
- Students and TAs should keep current-grade access first.
- Student-visible history can be added later after deciding whether old
  corrections, staff notes, source imports, and artifacts should be visible.

History views should include event metadata and previous/current grade facts,
but not require clients to reconstruct current state by hand.

## RLS and Grants

Preserve the current RLS shape for current-grade reads:

- quiz grades: students/TAs see only their own rows; faculty sees all.
- assignment grades: students/TAs see grades for assignment submissions visible
  to them, including team submissions; faculty sees all.
- snapshot grades: students/TAs see only their own rows; faculty sees all.

Apply RLS to the event tables too. Current views should remain owned and granted
in a way that keeps table RLS active, matching the current `api`-owned view
pattern unless implementation testing shows PostgreSQL 18 `security_invoker`
views are a cleaner fit in this codebase.

Write policies should remain faculty-only at the API boundary. Event tables
should deny direct writes to student and TA roles.

## Migration Plan

1. Add event tables and append-only grants/RLS.
2. Backfill one `recorded` event from each existing row in
   `data.assignment_grade`, `data.quiz_grade`, and `data.grade`.
3. Add current views and verify they match the old tables row-for-row.
4. Repoint `api.assignment_grades`, `api.quiz_grades`, and `api.grades` to the
   current views.
5. Add `INSTEAD OF` triggers for compatible `INSERT`, `UPDATE`, and `DELETE`.
6. Repoint distribution views to current views.
7. Update Python/admin grade writers to use the API view or dedicated RPC.
8. Rename legacy mutable tables to `*_legacy` or leave them read-only for one
   release. Do not keep two writable grade sources.
9. Add history API views after current read/write compatibility is proven.

The backfill step should be idempotent only within a controlled migration
transaction. Do not run a production backfill twice without a guard such as an
import marker or empty-event-table assertion.

## Test Plan

Database tests:

- Event tables reject direct student/TA writes and permit only faculty/API
  append paths.
- Inserting through `api.assignment_grades`, `api.quiz_grades`, and `api.grades`
  appends exactly one event and returns a current row with the existing shape.
- Updating through those API views appends a new event, preserves the old event,
  and changes only the current view result.
- Deleting through those API views appends a void event, preserves older events,
  and removes the row from the current view.
- Point-range, foreign-key, and default-filling behavior matches current tests.
- `created_at` remains first-event time and `updated_at` advances on correction.
- Distribution views count only current non-voided grades.
- RLS tests cover anonymous denial, student own/team visibility, TA visibility,
  faculty visibility, and faculty-only writes.

REST tests:

- Existing `/assignment_grades` and `/quiz_grades` list tests keep passing.
- Faculty `POST`, `PATCH`, and `DELETE` through PostgREST append events.
- Student requests cannot mutate current grades or event history.

Client/admin tests:

- Elm decoders continue to parse current grade payloads without code changes.
- Python quiz grading reruns append corrections without losing prior values.
- Course-admin grade imports can attach `source`, `reason`, `import_id`, and
  optional artifact metadata.

## Open Decisions

- Whether to keep `REAL` in event tables for exact compatibility or move events
  to `NUMERIC` while casting current views back to `REAL`. Exact compatibility
  argues for `REAL`; grade arithmetic correctness argues for `NUMERIC`.
- Whether `voided` rows should make current views hide the grade entirely or
  expose an explicit null/void status. Hiding best preserves current semantics.
- How much grade-event history students should see.
- How to derive `created_by_user_id` for non-human admin/import jobs.
- Whether grade exceptions should get their own append-only history in the same
  migration or in a follow-up. They affect grade access and due dates, but are
  not grade values.
- How to pair this work with team membership history. Current assignment grade
  visibility depends on present team membership, which can make historical
  access drift after team changes.

## Acceptance Mapping

- Design doc exists before migration implementation: this document.
- Current grade reads remain simple for Elm/admin consumers: keep existing API
  resource names and one-current-row views.
- Grade changes are not destructive: faculty/admin writes append events rather
  than mutating or deleting prior facts.
- Tests prove old grades remain queryable and current grades are correct: the
  test plan includes event-preservation, current-view, distribution, RLS, REST,
  and admin rerun coverage.
